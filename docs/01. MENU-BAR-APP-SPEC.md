# Meet-When macOS Menu Bar App вҖ” Design Specification

## 1. Overview

A native macOS menu-bar app (Swift/SwiftUI) that provides quick access to Meet-When scheduling features: view today's appointments, approve/reject pending bookings, and launch conference calls вҖ” all without opening a browser.

## 2. Prerequisite: JSON API Layer

The current Meet-When backend serves HTML (HTMX). The menu-bar app needs a **new JSON API namespace** added to the Go server.

### 2.1 New API Routes (`/api/v1/...`)

All routes require authentication via either:
- **Session cookie** (existing mechanism) вҖ” for browser-based flows
- **Bearer token** (`Authorization: Bearer <session_token>`) вҖ” for the native app

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/api/v1/auth/login` | Email + password вҶ’ `{ token, host, tenant }` |
| `POST` | `/api/v1/auth/login/select-org` | Complete multi-org login вҶ’ `{ token, host, tenant }` |
| `POST` | `/api/v1/auth/logout` | Invalidate session token |
| `GET` | `/api/v1/me` | Current host profile + tenant info |
| `GET` | `/api/v1/bookings?status=&date=&include_archived=` | List bookings with filters |
| `GET` | `/api/v1/bookings/today` | Today's confirmed bookings for the host |
| `GET` | `/api/v1/bookings/pending` | Pending bookings awaiting approval |
| `GET` | `/api/v1/bookings/{id}` | Single booking details |
| `POST` | `/api/v1/bookings/{id}/approve` | Approve a pending booking |
| `POST` | `/api/v1/bookings/{id}/reject` | Reject with `{ reason }` body |
| `POST` | `/api/v1/bookings/{id}/cancel` | Cancel with `{ reason }` body |

### 2.2 JSON Response Shapes

**Login Response:**
```json
{
  "token": "abc123...",
  "requires_org_selection": false,
  "orgs": null,
  "host": {
    "id": "uuid",
    "name": "Jane Doe",
    "email": "jane@example.com",
    "timezone": "America/New_York",
    "smart_durations": true
  },
  "tenant": {
    "id": "uuid",
    "name": "Acme Corp",
    "slug": "acme"
  }
}
```

If `requires_org_selection` is `true`, `orgs` contains `[{ id, name, slug, selection_token }]` and the client must call `/api/v1/auth/login/select-org` with the chosen `selection_token`.

**Booking Object:**
```json
{
  "id": "uuid",
  "template_name": "30-min Intro Call",
  "status": "confirmed",
  "start_time": "2026-03-30T14:00:00Z",
  "end_time": "2026-03-30T14:30:00Z",
  "duration": 30,
  "invitee_name": "Bob Smith",
  "invitee_email": "bob@example.com",
  "invitee_timezone": "Europe/London",
  "conference_link": "https://meet.google.com/abc-defg-hij",
  "location_type": "google_meet",
  "requires_approval": true,
  "created_at": "2026-03-29T10:00:00Z"
}
```

### 2.3 Auth Middleware Change

Add Bearer token support to the existing `RequireAuth` middleware:

```go
func (m *Middleware) RequireAuth(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        var token string
        // 1. Check Authorization header first (native app)
        if auth := r.Header.Get("Authorization"); strings.HasPrefix(auth, "Bearer ") {
            token = strings.TrimPrefix(auth, "Bearer ")
        }
        // 2. Fall back to session cookie (browser)
        if token == "" {
            if cookie, err := r.Cookie("session"); err == nil {
                token = cookie.Value
            }
        }
        // ... existing validation logic using token ...
    })
}
```

No new token type needed вҖ” reuse the existing 32-byte session tokens stored in the `sessions` table.

| Action | Behavior |
|--------|----------|
| **Approve** | Calls `POST /api/v1/bookings/{id}/approve`, shows checkmark animation, moves to Today if today |
| **Reject** | Shows a small text field for reason, then calls `POST /api/v1/bookings/{id}/reject` |

| **Open Dashboard** | Opens `{baseURL}/dashboard` in browser |
| **Sign Out** | Calls logout API, clears Keychain token, returns to login |
| **Click menu bar icon** | Toggles the popover |

## 5. Polling & Notifications

| Event | Mechanism |
|-------|-----------|
| Refresh bookings | Poll `GET /api/v1/bookings/today` + `/api/v1/bookings/pending` every **60 seconds** (configurable) |
| New pending booking | Compare pending count before/after poll; if increased, fire `UNUserNotificationCenter` notification: *"New booking request from {name}"* |
| Upcoming meeting (5 min) | Local scheduled notification based on `start_time` of confirmed bookings. Actionable: "Join" button opens conference link |
| Session expired | API returns 401 вҶ’ clear token, show login view |

## 6. Security Considerations

| Concern | Mitigation |
|---------|-----------|
| Token storage | macOS Keychain via `SecItemAdd`/`SecItemCopyMatching`, never `UserDefaults` |
| Token in memory | Store in `@Published` property, cleared on logout |
| HTTPS enforcement | `APIClient` rejects non-HTTPS URLs (allow override for `localhost` in dev) |
| Session lifetime | Reuses server's 7-day session expiry; app re-authenticates on 401 |
| Self-signed certs | Optional toggle in settings to trust custom CA (via `URLSessionDelegate`) |

## 7. Implementation Phases

### Phase 1 вҖ” API Layer (Go backend)
1. Add Bearer token support to `RequireAuth` middleware
2. Create `internal/handlers/api_v1.go` with JSON endpoints
3. Wire routes under `/api/v1/` in `main.go`
4. Add integration tests for all API endpoints

### Phase 2 вҖ” Core macOS App
1. `MenuBarExtra` shell with login/logout flow
2. `APIClient` with Keychain storage
3. Today's appointments list (read-only)
4. Join conference button

### Phase 3 вҖ” Pending Approvals
1. Pending bookings section with approve/reject
2. Reject reason input
3. Badge count on menu bar icon

### Phase 4 вҖ” Notifications & Polish
1. Background polling with configurable interval
2. macOS notifications for new pending bookings
3. 5-minute pre-meeting reminder notification
4. "Open Dashboard" deep link
5. Sparkle or auto-update mechanism

Files to modify/create in the Go codebase:

| File | Change |
|------|--------|
| `internal/middleware/auth.go` | Add `Authorization: Bearer` token extraction |

# Sign in with Apple Spec

Related GitHub issue: `#10` ("Google and Apple auth")

Issue status as of April 26, 2026:
- Google is complete.
- Apple is not implemented.

## Goal

Add Apple as a first-class auth provider for account signup and login, with behavior that matches the existing Google flow as closely as possible.

The current codebase already supports:
- Web signup with Google
- Web login with Google
- Native menu bar login with Google
- Auto-linking an OAuth identity to an existing account by email
- Multi-org selection after login

Apple should preserve those same product behaviors where possible.

## Current State

Google auth is implemented in several layers:

- Routing in `cmd/server/main.go`
- Web auth handlers in `internal/handlers/auth.go`
- Native app OAuth entrypoint in `internal/handlers/api_v1.go`
- OAuth business logic in `internal/services/auth.go`
- Host persistence in `internal/models/models.go`, `internal/repository/repository.go`, and migration `009_add_google_identity`
- Login and registration UI in `templates/pages/login.html`, `templates/pages/register.html`, and `templates/pages/register_google.html`
- Native menu bar login button in `MeetWhenBar/MeetWhenBar/Views/LoginView.swift`

This matters because Apple is not just "one more button". The Google implementation already spans web, server callbacks, DB persistence, signup conversion tracking, and the native client.

## Recommended Scope

Use a two-phase rollout.

### Phase 1: Web parity

Ship Apple signup and login for the web app only.

This includes:
- Apple button on web register page
- Apple button on web login page
- Apple OAuth start route
- Apple callback route
- Apple account completion form for first-time signup
- Auto-linking by email
- Multi-org selection
- Session creation and redirect behavior identical to Google

### Phase 2: Native menu bar parity

Add Sign in with Apple to the macOS menu bar app.

This should be treated as a separate follow-up even if implemented soon after Phase 1, because:
- native Apple auth uses `AuthenticationServices`, not the browser redirect flow used for Google
- native Apple auth returns credentials differently from the web flow
- Apple account grouping must be set up correctly if web and native should resolve to the same user identity

If the goal is only to close issue `#10` for the web app, Phase 1 is enough. If the goal is true product parity with existing Google support across the repo, Phase 2 should be included.

## Product Behavior To Match

Apple login/signup should match the current Google UX:

- Existing password account + same email:
  link Apple identity to the existing host record and log the user in
- Existing multi-org account + same email:
  link Apple identity to all matching hosts and show org selection
- No existing account during login:
  show "no account found, sign up first"
- First-time signup:
  complete OAuth, then collect organization name, slug, timezone, and editable display name
- Signup conversion tracking:
  preserve `ref=booking:...` across the Apple signup flow and mark conversion registered after completion

## Apple-Specific Constraints

These are the main differences from Google that drive the implementation shape:

1. Apple web auth returns the callback as `POST` when scopes are requested.
If we request `name` and `email`, Apple requires `response_mode=form_post`, so the callback route must accept `POST`, not just `GET`.

2. Apple does not have a Google-style userinfo endpoint for this flow.
The canonical identity comes from the `id_token` claims and the one-time `user` payload returned during first authorization.

3. The user's name is only provided the first time they authorize.
That means the server must persist the returned name immediately or preserve it in a signed cookie/session before rendering the completion form.

4. Apple users may choose a relay email.
Email can look like `...@privaterelay.appleid.com`. That should be treated as the user's real login email for this provider, not rejected as suspicious.

5. Apple local development is more restrictive than Google.
Apple's registered `redirect_uri` must be HTTPS, must include a domain, and cannot be `localhost` or a raw IP. Local testing will need a real HTTPS dev domain or a tunnel.

## Recommended Data Model

For the smallest delta from the current Google approach, extend `hosts` with Apple-specific columns:

- `apple_id`
- `apple_email`

Recommended migration behavior:
- nullable columns
- unique index on `apple_id` where not null
- no change to existing password accounts

This mirrors the current Google shape and lets the team ship quickly without introducing a new `identities` table.

### Alternative

A generic `host_identities` table would be cleaner long term:
- `id`
- `host_id`
- `provider`
- `provider_subject`
- `provider_email`
- timestamps

That is a better design if more providers are expected soon. It is more work because it touches every Google path too. For just Google + Apple, the column-based approach is the pragmatic choice.

## Server-Side Code Changes

### 1. Config

Files:
- `internal/config/config.go`
- `README.md`
- `.env.example` if present in the deployment setup

Add Apple config under `OAuth`:
- `APPLE_TEAM_ID`
- `APPLE_CLIENT_ID`
- `APPLE_KEY_ID`
- `APPLE_PRIVATE_KEY`
- `APPLE_REDIRECT_URL`

Notes:
- For web, `APPLE_CLIENT_ID` should be the Apple Services ID.
- If native parity is added later, the native app bundle ID and web Services ID must be grouped under the same Sign in with Apple setup in Apple Developer.
- `APPLE_PRIVATE_KEY` can be stored as PEM text in an env var, but operationally a file path is safer. If the team prefers that, use `APPLE_PRIVATE_KEY_PATH` instead.

### 2. Database / model / repository

Files:
- `migrations/*.sql`
- `migrations/sqlite/*.sql`
- `internal/models/models.go`
- `internal/repository/repository.go`

Changes:
- add `AppleID` and `AppleEmail` fields to `models.Host`
- update host insert/select scans
- add `GetByAppleID(ctx, appleID string) ([]*models.Host, error)`
- add `LinkAppleIdentity(ctx, hostID, appleID, appleEmail string) error`

### 3. Routes

File:
- `cmd/server/main.go`

Add web auth routes similar to Google:
- `GET /auth/apple/signup`
- `GET /auth/apple/login`
- `POST /auth/apple/auth-callback`
- `GET /auth/register/complete-apple`
- `POST /auth/register/complete-apple`

Important:
- the callback must support `POST`
- if the team wants to be defensive, allow both `GET` and `POST`, but the Apple path with scopes should expect `POST`

For Phase 2 native parity, also add:
- `POST /api/v1/auth/apple`

That native endpoint will not mirror Google exactly. It should accept the Apple credential payload from the macOS app and complete server-side verification/session creation.

### 4. Auth service

File:
- `internal/services/auth.go`

Add Apple-specific types and flows.

Recommended new types:
- `AppleUserInfo`
- `appleTokenResponse`
- `appleIDTokenClaims`

Recommended new methods:
- `GetAppleAuthURL(flow string) (authURL string, nonce string, err error)`
- `HandleAppleCallback(code, idToken, state, expectedNonce string, rawUser string) (*AppleUserInfo, string, error)`
- `RegisterWithApple(ctx, appleID, email, name, tenantName, tenantSlug, timezone string) (string, error)`
- `LoginWithApple(ctx, appleID, email string) (*GoogleLoginResult, error)` or introduce a provider-neutral result type
- `exchangeAppleAuthCode(code string) (*appleTokenResponse, error)`
- `generateAppleClientSecret() (string, error)`
- `verifyAppleIDToken(idToken, expectedNonce string) (*appleIDTokenClaims, error)`

Implementation details:
- build the authorization URL against `https://appleid.apple.com/auth/authorize`
- request `scope=name email`
- use `response_type=code id_token`
- use `response_mode=form_post`
- include `state`
- include `nonce`
- exchange the code at `https://appleid.apple.com/auth/token`
- generate the `client_secret` JWT with ES256 using Apple key ID, team ID, and the private key
- verify the returned `id_token` signature and claims

Claims that must be validated:
- signature using Apple's public keys
- `iss == https://appleid.apple.com`
- `aud == APPLE_CLIENT_ID`
- `exp` in the future
- `nonce` matches the one stored in the cookie

Data to persist/use:
- `sub` as `apple_id`
- `email`
- first-login name from the callback `user` payload when present

### 5. Web handlers

File:
- `internal/handlers/auth.go`

Add Apple equivalents of the Google handlers:
- `AppleSignupStart`
- `AppleLoginStart`
- `AppleAuthCallback`
- `CompleteAppleRegisterPage`
- `CompleteAppleRegister`
- cookie signing/reading helpers for Apple profile state

Behavior should stay aligned with Google:
- store nonce in an `HttpOnly` cookie
- preserve `ref` during signup
- store Apple profile in a signed cookie for the completion page
- set session cookie on success
- reuse signup conversion tracking after registration

Important difference:
- the Apple callback handler should parse `application/x-www-form-urlencoded` `POST` data
- it must read `code`, `id_token`, `state`, and optional `user`
- cancellation/error handling should read the posted error fields as well

### 6. Templates

Files:
- `templates/pages/login.html`
- `templates/pages/register.html`
- new `templates/pages/register_apple.html`, or reuse the existing Google completion template with provider-neutral copy

Changes:
- add "Continue with Apple" on login
- add "Sign up with Apple" on register
- render the button in a way that matches Apple's branding rules
- update completion copy from "Google Account" to either:
  - a new Apple-specific page, or
  - a shared OAuth completion page that displays the provider name dynamically

Recommendation:
- create a provider-neutral completion template now, because Google and Apple require nearly identical final-step forms

### 7. Native menu bar app

Files:
- `MeetWhenBar/MeetWhenBar/Views/LoginView.swift`
- `MeetWhenBar/MeetWhenBar/ViewModels/AppViewModel.swift`
- `MeetWhenBar/MeetWhenBar/Info.plist`
- Xcode project capability settings

Phase 2 changes:
- add `SignInWithAppleButton` to the login view
- use `AuthenticationServices`
- request scopes `.fullName` and `.email`
- send `authorizationCode`, `identityToken`, `user`, `email`, `fullName`, and `state` to a new server API
- persist the returned session token exactly like the current Google/native callback flow

Important native design decision:

Do not force native Apple login through the browser redirect flow.

That would work, but it wastes the platform-native capability already available on macOS. The correct native implementation is:
- native app performs Apple sign-in locally
- native app sends the Apple credential bundle to the server
- server verifies identity and creates the Meet When session

### 8. API v1 native flow

File:
- `internal/handlers/api_v1.go`

For Phase 2, add a dedicated Apple endpoint, for example:
- `POST /api/v1/auth/apple`

Request body should include:
- `authorization_code`
- `identity_token`
- `user`
- `email`
- `full_name`
- `state`

Server behavior:
- verify token/code with Apple
- upsert/link account using same rules as web
- return `{ token, host, tenant }` or the existing redirect/session response shape

Recommendation:
- for native Apple, return JSON directly instead of using the custom URL scheme redirect
- keep the custom URL scheme only for browser-based Google login

That gives cleaner app logic and removes an unnecessary loop through the browser.

## Process / Apple Developer Setup

These steps are required before code can work in any environment.

### Web setup

1. Have an Apple Developer account with Sign in with Apple available.
2. Enable Sign in with Apple on an App ID and set the primary app.
3. Create a Services ID for the web app.
4. Register the production domain and callback URL for the Services ID.
5. Create a Sign in with Apple private key and download the `.p8` file.
6. Record:
   - Team ID
   - Key ID
   - Services ID
   - Redirect URL
7. Put the private key into the deployment secret store.

### Email relay setup

If Meet When sends transactional email to authenticated users, configure Apple's private email relay:

1. Register outbound sender domains or sender addresses in Apple Developer.
2. Ensure SPF is configured.
3. Prefer DKIM as well.
4. Validate that mail to `@privaterelay.appleid.com` addresses is accepted.

Without this, passwordless Apple users who choose Hide My Email may not receive mail reliably.

### Local and staging setup

Because Apple does not allow `localhost` or raw IP callback URLs for web Sign in with Apple:

1. Use a real HTTPS hostname for staging.
2. For local development, use a tunnel or HTTPS dev domain.
3. Register each environment callback URL explicitly in Apple Developer.

Recommendation:
- production first
- one stable staging URL second
- do not optimize for arbitrary localhost testing

### Native app setup

For Phase 2:

1. Add the Sign in with Apple capability in Xcode for `MeetWhenBar`.
2. Ensure the bundle ID is registered in Apple Developer.
3. Group the native app and web Services ID under the same primary App ID if you want the same Apple user relationship across surfaces.
4. Test credential state handling on macOS using `getCredentialState`.

## Suggested Implementation Order

1. Add config and secret-loading support.
2. Add DB migration, model fields, and repository methods.
3. Implement Apple auth service primitives:
   - auth URL
   - client secret generation
   - token exchange
   - ID token verification
4. Add web handlers and routes.
5. Add login/register buttons and the completion form.
6. Add tests for signup, login, linking, and multi-org behavior.
7. Validate production/staging Apple Developer configuration.
8. Add native menu bar support as a second PR or second milestone.

## Testing Plan

### Unit / integration tests

Add tests for:
- Apple callback state/nonce validation
- Apple callback error/cancel paths
- ID token claim validation failures
- signup with Apple creates tenant, host, working hours, session, and audit log
- login with Apple by existing `apple_id`
- login with Apple by existing email auto-links the Apple identity
- multi-org selection when one Apple identity maps to multiple hosts
- signup conversion `ref` survives the Apple flow

### Manual QA

Test these scenarios:
- new user signs up with Apple shared email
- new user signs up with Apple relay email
- existing password user logs in with Apple and gets linked
- multi-org existing user logs in with Apple and sees org selection
- Apple login cancel flow
- expired nonce cookie
- replayed or invalid `state`
- invalid or expired `id_token`
- mail delivery to a relay email address

### Native QA

For Phase 2:

- revoked Apple credentials
- app relaunch after successful native sign-in

## Risks And Edge Cases

### 1. Name is one-time data

If the server fails between callback receipt and persistence, the user's name may be lost. Mitigation: persist or sign-cookie the `user` payload immediately.

### 2. Hide My Email affects identity linking

If a user previously registered with a real email and later uses Apple relay email, email-based auto-link will not match. That is expected behavior. In that case the Apple login should behave like a new identity unless the account was already linked.

### 3. Local dev friction

Apple setup is significantly less forgiving than Google. Expect staging-first testing.

### 4. Native and web identity consistency

If the Apple Developer configuration is not grouped correctly, the same user may not map cleanly across native and web surfaces. This is a setup risk, not only a code risk.

### 5. Provider-specific duplication

The current auth code is Google-specific by name. Adding Apple by copy/paste is fast but increases maintenance cost. Some small refactoring is worth doing now:
- shared provider-neutral signed profile cookie helper
- shared OAuth completion template
- shared provider-neutral login result type if Apple and Google logic converge

### Phase 1: Web parity

Roughly:
- 1 migration
- 4 to 6 files with real auth logic changes
- 2 to 3 templates touched or added
- 1 config update
- test additions

Expected effort:
- about 1 to 2 focused engineering days if Apple Developer access and secrets are already available
- longer if the team has to set up Apple Developer assets, email relay, and staging domains from scratch

### Phase 2: Native parity

Expected effort:
- about 0.5 to 1.5 additional days

This depends mostly on:
- whether the team accepts a dedicated JSON native endpoint
- how much credential-state handling they want in the menu bar app

## Recommendation

Implement Apple in two PRs:

1. Web Apple auth parity
2. Native menu bar Apple auth parity

That keeps the first PR tightly scoped to issue `#10`'s core ask while avoiding a larger, riskier cross-surface rollout.
